# 版本五：设备与离线同步版

## 版本目标

在 v4 WebSocket 实时通信基础上，落地用户设备记录和离线同步规则。

v5 保存设备信息和推送 token，但不真正接入 APNs、FCM 或厂商推送。

## 前置基础

- v1 已实现用户和 JWT 鉴权
- v3 已实现历史消息和未读数
- v4 已实现在线用户 WebSocket 实时投递

## 接口设计

### 上报设备

```text
POST /api/devices
```

请求体：

```json
{
  "device_id": "web-001",
  "platform": "web",
  "push_token": ""
}
```

成功响应：

```json
{
  "code": 200,
  "data": {
    "id": 1,
    "user_id": 1,
    "device_id": "web-001",
    "platform": "web",
    "push_token": "",
    "last_seen_at": "2026-06-29T12:00:00+08:00"
  },
  "message": "上报设备成功"
}
```

规则：

- `device_id` 必填
- `platform` 必须是 `web`、`ios`、`android`、`desktop`
- `push_token` 可为空
- 同一用户同一 `device_id` 重复上报时更新已有记录
- 每次上报都更新 `last_seen_at`

### 获取设备列表

```text
GET /api/devices
```

成功响应：

```json
{
  "code": 200,
  "data": [
    {
      "device_id": "web-001",
      "platform": "web",
      "push_token": "",
      "last_seen_at": "2026-06-29T12:00:00+08:00"
    }
  ],
  "message": "获取设备列表成功"
}
```

规则：

- 只返回当前用户自己的设备
- 按 `last_seen_at` 倒序返回

### 删除设备

```text
DELETE /api/devices/{device_id}
```

成功响应：

```json
{
  "code": 200,
  "data": null,
  "message": "删除设备成功"
}
```

规则：

- 只能删除当前用户自己的设备
- 删除不存在的设备可以返回成功，保证接口幂等

## 离线同步规则

v5 的离线能力不依赖推送服务，核心是“消息先入库，客户端上线后补齐”。

客户端上线后建议流程：

1. 登录或刷新 token。
2. 调用 `POST /api/devices` 上报设备。
3. 建立 WebSocket。
4. 调用 `GET /api/conversations` 获取会话列表和未读数。
5. 对有未读数的会话调用 `GET /api/conversations/{conversation_id}/messages` 拉取历史消息。
6. 读完消息后调用 `POST /api/conversations/{conversation_id}/read` 标记已读。

## WebSocket 结合规则

- `/ws?token={jwt}&device_id={device_id}` 中的 `device_id` 可以用于更新设备 `last_seen_at`
- WebSocket 收到 `ping` 时可以更新连接的 `last_active_at`
- WebSocket 断开不删除设备，只影响在线状态
- 设备记录不等于在线连接，设备可以存在但不在线

## 核心流程

### 上报设备

1. 从 JWT 中获取当前用户 ID。
2. 校验 `device_id` 和 `platform`。
3. 查询 `(user_id, device_id)` 是否存在。
4. 已存在则更新 `platform`、`push_token`、`last_seen_at`。
5. 不存在则插入新设备记录。
6. 返回设备信息。

### 用户离线后同步

1. 用户离线期间，其他成员发送消息。
2. 服务端正常写入 `messages` 并更新会话最后消息。
3. 离线用户没有 WebSocket 连接，因此不实时收到 `message.deliver`。
4. 用户重新上线。
5. 客户端查询会话列表得到未读数。
6. 客户端按会话拉历史消息补齐。

## 边界规则

- v5 不调用真实推送服务
- v5 不保证客户端一定上报设备
- 删除设备不影响用户账号
- 删除设备不删除历史消息
- 同一用户可以有多个设备

## 错误处理

- `400`：参数校验失败
- `401`：未登录或登录已失效
- `403`：操作其他用户设备
- `500`：服务器内部错误

## 验证方式

1. 登录后调用 `POST /api/devices` 上报 web 设备。
2. 重复上报同一个 `device_id`，确认更新同一条记录。
3. 调用 `GET /api/devices`，确认能看到设备。
4. 删除设备后再次查询，确认设备消失。
5. 用户离线期间产生消息。
6. 用户上线后查询会话列表和历史消息，确认可以补齐离线消息。

## 本版本不包含

- APNs 推送
- FCM 推送
- 厂商推送
- 多端已读同步优化
- 在线状态公开展示
