# 版本三：消息与历史记录版

## 版本目标

在 v2 会话基础上，落地消息入库、历史消息分页、会话最后消息、会话级已读游标和未读数。

v3 使用 HTTP 接口发送消息，不实现 WebSocket 实时投递。

## 前置基础

- v1 已实现用户和 JWT 鉴权
- v2 已实现会话和成员关系
- 当前用户必须是会话 `active` 成员，才能发送、读取和标记已读

## 数据库文档

- [数据库表创建-消息](../数据库/数据库表创建-消息.md)

- 发送消息并持久化
- 分页拉取历史消息

## Go 模型设计

`models/message_model.go` 建议包含：

```go
const (
	MessageTypeText   = "text"
	MessageTypeImage  = "image"
	MessageTypeFile   = "file"
	MessageTypeSystem = "system"

	MessageStatusNormal   = "normal"
	MessageStatusRecalled = "recalled"
	MessageStatusDeleted  = "deleted"
)

type Message struct {
	ID             int64           `gorm:"column:id;primaryKey;autoIncrement" json:"id"`
	ConversationID int64           `gorm:"column:conversation_id" json:"conversation_id"`
	SenderID       int64           `gorm:"column:sender_id" json:"sender_id"`
	ClientMsgID    string          `gorm:"column:client_msg_id" json:"client_msg_id"`
	MessageType    string          `gorm:"column:message_type" json:"message_type"`
	Content        json.RawMessage `gorm:"column:content" json:"content"`
	Status         string          `gorm:"column:status" json:"status"`
	SentAt         time.Time       `gorm:"column:sent_at" json:"sent_at"`
	CreatedAt      time.Time       `gorm:"column:created_at" json:"created_at"`
}
```

需要实现的 model 方法：

- `func (Message) TableName() string`
- `func SendMessage(senderID int64, conversationID int64, clientMsgID string, messageType string, content json.RawMessage) (MessageDTO, error)`
- `func ListMessages(userID int64, conversationID int64, beforeID int64, limit int) (MessagePageDTO, error)`
- `func MarkConversationRead(userID int64, conversationID int64, lastReadMessageID int64) (ReadStateDTO, error)`
- `func FindMessageByID(messageID int64) (Message, error)`
- `func validateMessageContent(messageType string, content json.RawMessage) error`
- `func normalizeMessagePageLimit(limit int) int`

新增领域错误：

- `ErrMessageNotFound`
- `ErrInvalidMessageType`
- `ErrInvalidMessageContent`
- `ErrMessageForbidden`
- `ErrReadCursorInvalid`

## JSON 内容处理

`content` 字段不要用字符串拼接。controller 接收时使用 `json.RawMessage`：

```go
type SendMessageRequest struct {
	ClientMsgID string          `json:"client_msg_id" binding:"required"`
	MessageType string          `json:"message_type" binding:"required"`
	Content     json.RawMessage `json:"content" binding:"required"`
}
```

校验规则：

- `text`：解析为 `struct { Text string }`，`Text` 去空格后不能为空
- `image`：解析为包含 `URL string` 的结构，`URL` 不能为空
- `file`：解析为包含 `URL string`、`Name string`、`Size int64` 的结构，`URL` 不能为空
- `system`：普通 HTTP 发送接口不允许客户端提交

## 事务边界

`SendMessage` 必须使用事务：

1. 查询发送者是否是会话 `active` 成员。
2. 查询 `(sender_id, client_msg_id)` 是否已有消息。
3. 已有消息时直接返回，不再写入。
4. 写入 `messages`。
5. 更新 `conversations.last_message_id` 和 `last_message_at`。
6. 提交事务。

事务内任何一步失败，都不能留下只写消息但未更新会话最后消息的状态。

## 实现顺序

1. 在 MySQL 中执行 [数据库表创建-消息](../数据库/数据库表创建-消息.md) 的建表 SQL。

8. 新增 `controllers/message_controller.go`。
9. 修改 `routers/router.go` 注册消息路由。
10. 更新 `docs/openapi.json`。
11. 使用 curl 验证发送、历史、已读和未读数。

## 接口设计

### 发送消息

```text
POST /api/conversations/{conversation_id}/messages
```

请求体：

```json
{
  "client_msg_id": "client-001",
  "message_type": "text",
  "content": {
    "text": "hello"
  }
}
```

成功响应：

```json
{
  "code": 200,
  "data": {
    "id": 1001,
    "conversation_id": 1,
    "sender_id": 1,
    "client_msg_id": "client-001",
    "message_type": "text",
    "content": {
      "text": "hello"
    },
    "status": "normal",
    "sent_at": "2026-06-29T12:00:00+08:00"
  },
  "message": "发送消息成功"
}
```

规则：

- `client_msg_id` 必填
- `message_type` 必须是 `text`、`image`、`file`、`system` 之一
- 普通用户只能发送 `text`、`image`、`file`
- `system` 消息保留给服务端生成
- 文本消息必须包含非空 `content.text`
- 图片和文件消息必须包含非空 `content.url`
- 同一用户重复提交同一 `client_msg_id` 时返回已有消息

### 拉取历史消息

```text
GET /api/conversations/{conversation_id}/messages?before_id=1001&limit=20
```

参数说明：

- `before_id`：选填，拉取 ID 小于该值的消息；不传时从最新消息开始
- `limit`：选填，默认 `20`，最大 `100`

成功响应：

```json
{
  "code": 200,
  "data": {
    "items": [
      {
        "id": 1000,
        "conversation_id": 1,
        "sender_id": 2,
        "message_type": "text",
        "content": {
          "text": "hi"
        },
        "status": "normal",
        "sent_at": "2026-06-29T12:00:00+08:00"
      }
    ],
    "next_before_id": 1000,
    "has_more": true
  },
  "message": "获取历史消息成功"
}
```

规则：

- 只返回当前会话的消息
- 默认按 `id desc` 查询，再由客户端按需要展示
- `status = deleted` 的消息可以不返回
- `status = recalled` 的消息返回状态和基础信息，但内容可置空

### 标记已读

```text
POST /api/conversations/{conversation_id}/read
```

请求体：

```json
{
  "last_read_message_id": 1001
}
```

成功响应：

```json
{
  "code": 200,
  "data": {
    "conversation_id": 1,
    "last_read_message_id": 1001,
    "last_read_at": "2026-06-29T12:05:00+08:00"
  },
  "message": "标记已读成功"
}
```

规则：

- `last_read_message_id` 必须属于当前会话
- 不能把已读游标倒退到更小的消息 ID
- 标记已读只更新当前用户在当前会话中的成员记录

### 获取会话列表增强

```text
GET /api/conversations
```

v3 在 v2 响应基础上增加：

```json
{
  "last_message": {
    "id": 1001,
    "sender_id": 1,
    "message_type": "text",
    "content": {
      "text": "hello"
    },
    "sent_at": "2026-06-29T12:00:00+08:00"
  },
  "unread_count": 3
}
```

未读数计算：

```text
messages.id > conversation_members.last_read_message_id
```

如果 `last_read_message_id` 为空，则该会话内所有消息都算未读，但当前用户自己发送的消息可以不计入未读。

## 核心流程

### 发送消息

1. 从 JWT 中获取当前用户 ID。
2. 校验当前用户是会话 `active` 成员。
3. 校验 `client_msg_id`、`message_type` 和 `content`。
4. 查询 `(sender_id, client_msg_id)` 是否已有消息。
5. 已存在则直接返回已有消息。
6. 不存在则开启事务。
7. 写入 `messages`。
8. 更新 `conversations.last_message_id` 和 `conversations.last_message_at`。
9. 提交事务。
10. 返回新消息。

### 拉取历史消息

1. 校验当前用户是会话 `active` 成员。
2. 解析 `before_id` 和 `limit`。
3. 查询当前会话消息。
4. 按 `id desc` 返回一页。
5. 根据结果数量判断 `has_more`。

### 标记已读

1. 校验当前用户是会话 `active` 成员。
2. 校验 `last_read_message_id` 属于当前会话。
3. 查询当前成员已有 `last_read_message_id`。
4. 如果新游标更大，则更新游标和 `last_read_at`。
5. 如果新游标小于等于旧游标，保持旧值不变并返回当前状态。

## 边界规则

- 非会话成员不能发送消息
- 非会话成员不能拉取历史消息
- `client_msg_id` 只在同一发送者维度去重
- 发送消息失败时不能更新会话最后消息
- 标记已读不能影响其他成员
- v3 不生成逐条消息回执

## 错误处理

- `400`：参数校验失败
- `401`：未登录或登录已失效
- `403`：非会话成员或无权操作
- `404`：会话或消息不存在
- `409`：幂等冲突或消息状态冲突
- `500`：服务器内部错误

## 验证方式

1. 创建单聊或群聊。
2. 会话成员发送文本消息。
3. 重复使用同一个 `client_msg_id` 发送，确认只生成一条消息。
4. 拉取历史消息，确认能查到刚发送的消息。
5. 查询会话列表，确认返回最后消息和未读数。
6. 标记已读后再次查询会话列表，确认未读数变化。
7. 非成员发送消息，确认返回 `403`。

## 本版本不包含

- WebSocket 实时投递
- 离线推送
- 逐条消息已读详情
- 消息搜索
- 文件上传存储
