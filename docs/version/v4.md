# 版本四：WebSocket 实时通信版

## 版本目标

在 v3 消息能力基础上，增加 WebSocket 长连接，实现实时发送、确认和在线投递。

v4 只支持单进程内存连接管理，不支持多实例跨节点投递。

## 前置基础

- v1 已实现 JWT 鉴权
- v2 已实现会话和成员关系
- v3 已实现消息入库、历史消息和未读数
- WebSocket 发送消息复用 v3 的消息写入规则

## 数据库文档

本版本不新增数据库表，依赖：

- [数据库表创建-会话](../数据库/数据库表创建-会话.md)
- [数据库表创建-消息](../数据库/数据库表创建-消息.md)
- [数据库表关系](../数据库/数据库表关系.md)

## 本版本新增能力

- WebSocket 建连鉴权
- 心跳 `ping` / `pong`
- 客户端通过 WebSocket 发送消息
- 服务端返回消息确认 `message.ack`
- 服务端向在线成员投递 `message.deliver`
- 内存维护在线连接

## 依赖选择

当前 `go.mod` 尚未包含 WebSocket 库。v4 建议新增：

```text
github.com/gorilla/websocket
```

选择原因：

- Gin 项目中集成简单
- API 稳定，适合当前单进程连接管理
- 后续迁移到多实例时，连接管理逻辑仍可复用

## 代码落地清单

新增文件：

- `controllers/ws_controller.go`：处理 `/ws` 建连、token 校验和连接生命周期
- `models/ws_event.go`：定义 WebSocket 事件结构和事件类型常量
- `models/ws_hub.go`：实现单进程内存连接管理

修改文件：

- `models/message_model.go`：暴露可复用的消息发送函数，供 WebSocket 和 HTTP 共用
- `routers/router.go`：注册 `/ws`
- `go.mod`、`go.sum`：新增 WebSocket 依赖
- `docs/openapi.json`：OpenAPI 对 WebSocket 支持有限，可在 description 中补充 `/ws` 协议说明

## Go 结构设计

`models/ws_event.go` 建议包含：

```go
const (
	WSEventPing           = "ping"
	WSEventPong           = "pong"
	WSEventMessageSend    = "message.send"
	WSEventMessageAck     = "message.ack"
	WSEventMessageDeliver = "message.deliver"
	WSEventError          = "error"
)

type WSEvent struct {
	Type      string          `json:"type"`
	RequestID string          `json:"request_id,omitempty"`
	Payload   json.RawMessage `json:"payload,omitempty"`
}
```

`models/ws_hub.go` 建议包含：

```go
type WSConnection struct {
	ID           string
	UserID       int64
	DeviceID     string
	ConnectedAt  time.Time
	LastActiveAt time.Time
	Send         chan []byte
}

type WSHub struct {
	mu          sync.RWMutex
	connections map[int64]map[string]*WSConnection
}
```

需要实现的方法：

- `func NewWSHub() *WSHub`
- `func (h *WSHub) Add(conn *WSConnection)`
- `func (h *WSHub) Remove(userID int64, connectionID string)`
- `func (h *WSHub) UserConnections(userID int64) []*WSConnection`
- `func (h *WSHub) BroadcastToUsers(userIDs []int64, payload []byte)`
- `func (h *WSHub) Touch(userID int64, connectionID string)`

## Controller 设计

`controllers/ws_controller.go` 建议包含：

```go
type WSController struct {
	JWT models.JWTConfig
	Hub *models.WSHub
}
```

需要实现的方法：

- `func (ctl WSController) Connect(c *gin.Context)`
- `func (ctl WSController) readLoop(user models.User, conn *websocket.Conn, wsConn *models.WSConnection)`
- `func (ctl WSController) writeLoop(conn *websocket.Conn, wsConn *models.WSConnection)`
- `func (ctl WSController) handleEvent(user models.User, event models.WSEvent, wsConn *models.WSConnection)`
- `func (ctl WSController) handleMessageSend(user models.User, event models.WSEvent, wsConn *models.WSConnection)`

建连时不能复用 `AuthRequired`，因为 token 在 query 中。应直接调用 `middlewares.VerifyToken`，再用 `models.FindUserByID` 刷新用户状态。

## 路由落地

`routers/router.go` 中创建全局 hub，并注册：

```go
wsHub := models.NewWSHub()
wsController := controllers.WSController{JWT: cfg.JWT, Hub: wsHub}

engine.GET("/ws", wsController.Connect)
```

## 事件处理落地

`message.send` 的处理必须复用 v3 的 `models.SendMessage`：

1. 解析 payload。
2. 调用 `models.SendMessage`。
3. 成功后向当前连接写入 `message.ack`。
4. 查询会话 active 成员。
5. 组装 `message.deliver`。
6. 调用 hub 投递给在线成员。

不要在 WebSocket controller 中重新写一套消息入库逻辑。

## 实现顺序

1. 执行 `go get github.com/gorilla/websocket` 增加 WebSocket 依赖。
2. 新增 `models/ws_event.go`，定义事件类型和统一事件结构。
3. 新增 `models/ws_hub.go`，实现单进程连接管理。
4. 新增 `controllers/ws_controller.go`，先实现建连鉴权和连接注册/移除。
5. 实现 `ping` / `pong`。
6. 实现 WebSocket `message.send` payload 解析。
7. 复用 v3 `models.SendMessage` 写入消息。
8. 实现 `message.ack`。
9. 查询会话 active 成员并实现 `message.deliver` 在线投递。
10. 修改 `routers/router.go` 注册 `/ws`。
11. 使用两个 WebSocket 客户端验证 A 发消息、B 实时收到。

## WebSocket 地址

```text
GET /ws?token={jwt}&device_id={device_id}
```

参数说明：

- `token`：必填，v1 登录或注册返回的 JWT
- `device_id`：选填，客户端设备标识；v5 设备版本落地后可与设备表关联

建连规则：

- token 无效或过期时拒绝连接
- 用户被禁用时拒绝连接
- 同一用户可以有多个在线连接
- 连接断开后必须从内存连接管理器中移除

## 事件格式

客户端和服务端统一使用 JSON：

```json
{
  "type": "event.name",
  "request_id": "req-001",
  "payload": {}
}
```

字段说明：

- `type`：事件类型
- `request_id`：请求 ID，客户端请求类事件建议传入，服务端确认时原样返回
- `payload`：事件数据

## 客户端事件

### ping

```json
{
  "type": "ping",
  "request_id": "req-ping-001"
}
```

服务端返回：

```json
{
  "type": "pong",
  "request_id": "req-ping-001",
  "payload": {
    "server_time": "2026-06-29T12:00:00+08:00"
  }
}
```

### message.send

```json
{
  "type": "message.send",
  "request_id": "req-001",
  "payload": {
    "conversation_id": 1,
    "client_msg_id": "client-001",
    "message_type": "text",
    "content": {
      "text": "hello"
    }
  }
}
```

规则：

- `conversation_id` 必填
- `client_msg_id` 必填
- `message_type` 和 `content` 校验规则与 v3 HTTP 发送消息一致
- 发送者必须是会话 `active` 成员

## 服务端事件

### message.ack

服务端收到 `message.send` 并成功入库后，向发送连接返回确认。

```json
{
  "type": "message.ack",
  "request_id": "req-001",
  "payload": {
    "message_id": 1001,
    "conversation_id": 1,
    "client_msg_id": "client-001",
    "sent_at": "2026-06-29T12:00:00+08:00"
  }
}
```

### message.deliver

服务端向会话内在线成员投递消息。

```json
{
  "type": "message.deliver",
  "payload": {
    "message_id": 1001,
    "conversation_id": 1,
    "sender_id": 2,
    "message_type": "text",
    "content": {
      "text": "hello"
    },
    "sent_at": "2026-06-29T12:00:00+08:00"
  }
}
```

投递规则：

- 投递给会话内 `active` 成员的在线连接
- 同一用户多端在线时，每个连接都可以收到
- 是否投递给发送者自己的其他连接，由服务端统一决定；建议投递给发送者除当前连接外的其他连接
- 离线成员不实时投递，后续通过历史消息补齐

### error

事件处理失败时返回：

```json
{
  "type": "error",
  "request_id": "req-001",
  "payload": {
    "code": 400,
    "message": "参数校验失败"
  }
}
```

## 内存连接管理

v4 可以使用单进程内存结构：

```text
user_id -> connection_id -> connection
```

连接信息至少包含：

- `connection_id`
- `user_id`
- `device_id`
- `connected_at`
- `last_active_at`

## 核心流程

### 建立连接

1. 客户端携带 `token` 请求 `/ws`。
2. 服务端校验 token。
3. 服务端查询用户状态。
4. 升级 HTTP 为 WebSocket。
5. 生成连接 ID。
6. 写入内存连接管理器。
7. 等待读写事件。

### WebSocket 发送消息

1. 服务端读取客户端事件。
2. 根据 `type = message.send` 解析 payload。
3. 复用 v3 消息发送逻辑写入数据库。
4. 向发送连接返回 `message.ack`。
5. 查询会话 active 成员。
6. 找出在线成员连接。
7. 向在线连接发送 `message.deliver`。

### 连接断开

1. 读写异常或客户端关闭连接。
2. 从内存连接管理器删除连接。
3. 如果用户没有其他连接，则该用户视为离线。

## 边界规则

- v4 不保证消息一定实时送达离线用户
- v4 不支持跨服务实例投递
- v4 不要求服务端保存 WebSocket 事件日志
- 客户端收到 `message.ack` 表示消息已入库
- 客户端没收到 `message.ack` 时，可以使用同一 `client_msg_id` 重试

## 错误处理

- token 无效：拒绝连接或关闭连接
- JSON 格式错误：返回 `error`
- 未知事件类型：返回 `error`
- 发送消息失败：返回 `error`，不返回 `message.ack`
- 投递某个连接失败：关闭该连接并从连接管理器移除

## 验证方式

1. 使用有效 token 建立 WebSocket，确认连接成功。
2. 使用无效 token 建立 WebSocket，确认连接失败。
3. 客户端发送 `ping`，确认收到 `pong`。
4. 用户 A 和用户 B 建立连接。
5. 用户 A 发送 `message.send`。
6. 用户 A 收到 `message.ack`。
7. 用户 B 收到 `message.deliver`。
8. 用户 B 断开后，用户 A 再发送消息，确认消息仍入库。
9. 用户 B 重新上线后，通过 v3 历史消息接口拉取离线期间消息。

## 本版本不包含

- Redis Pub/Sub
- 多实例 WebSocket 投递
- 移动端离线推送
- 消息送达回执
- 逐条已读回执
